SASE Event Tracker

HTML, CSS, Javascript, MongoDB

Inspiration

I was an officer for a UCSD club, SASE, which needed a better way to track our events. We would lose track of which weeks we had events and what types of events that we had in a single week (social, professional, etc.) One problem we faced was having too many professional events crammed in a week, which left no space for any other events. I conceptualized this app with another board member, but ended up implementing it myself after she broke her leg.

Implementation

The first part to implement was having events displayed on the web page. Since UCSD students talk about which "week" it is rather than the actual date, we felt that it would be beneficial to have events listed per week. We came up with a button for each week, which opened up a list view of events for that week. Then, I added an "Add event page" that included relevant information such as date, week, location, etc. Another issue that came up was trying to distinguish between different events (social, professional, etc.) so we decided to color code them. This way we could easily figure out whether we had too many events per week.

Improvement

After showing the app to some of the other board members, I received feedback regarding making the app more user friendly. Some wanted to see a calendar list too, a way to see the most recent added events, and notifications when someone added a new event.

Deja Moo

HTML, CSS, Javascript, MongoDB, Express.js

Brainstorming

For my Human-Computer Interaction Studio class, I worked on a quarter-long project with three other classmates to create a web app focused on location tracking. We started off with a brainstorming assignment where we came up with eight possible app ideas and then created story boards for them. Our ideas included a warring kingdoms game, 3D visualizations of the busiest places, and even a Tinder match finder! Yet, nothing seemed to click. Then, during our meeting we realized we would all appreciate being able to know when the library was full, about flash sales, and other events around the community. Along with our professor’s love for cows, we ended up with Deja Moo.

First Iteration

For our first iteration, we started off with a very basic approach: pin generic cows onto a map with the ability to add comments. However, this would lead to a very bland experience. There was no interaction between different users besides posting and the app was not memorable. At this point, we were struggling with our own brand image as well as figuring out the best way to allow users to add comments to each cow.

Second Iteration

For our second iteration, one of my group members drew his own cows which gave the app a more fun personality as well as allowed users to differentiate between different types of cows (events, sales, food, etc.). I implemented the upvoting system which would allow users to recognize important posts and hooked the web app to a MongoDB database. At this point, the app has a clearer direction but lacks proper organization of icons, making for a confusing user experience.

Final Iteration

In our final iteration for the class, we decided to create a login system that would track users’ own cows, allow them to delete these cows, and used a blue border to signify one’s own cows. Users are only allowed to drop a cow pin within a certain radius of their current location and cows that are too close to each other cluster together. I added a quick tutorial on the app, a search bar for users that were curious about cows in other locations, and a filter for different cow types. Overall, I was responsible for much of the app’s functionality and suggestions for reorganizing the layout of our app.
